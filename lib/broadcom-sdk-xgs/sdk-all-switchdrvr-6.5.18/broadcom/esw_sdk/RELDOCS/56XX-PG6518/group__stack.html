◆ bcm_stk_modid_config_add()

int bcm_stk_modid_config_add ( int  unit,
bcm_stk_modid_config_t *  modid 
)

Adds a desired module local to the system.

Adds a desired module local to the system. These set of APIs are applicable on BCM5645x, BCM5626x, BCM5646x, BCM5686x, BCM5656x, and BCM5676x devices. Modules added with this API can be used for subtended ports and nothing precludes us from creating modules that are non-consecutive for this purpose. It is worth nothing that in stacking applications where the requirement is to have consecutive module Ids, this non-consecutive module Id assignment might cause issues and hence care should be taken when assiging thse module Ids. Also, the module Ids assigned to subtended modules may not be reused within the same stacking system. The maximum number of these modules supported on BCM5686x, BCM5656x, and BCM5676x devices is five and on BCM5645x, BCM5626x BCM5646x it is three. Furthermore on BCM5645x, BCM5626x and BCM5646x if the number of ports on modules being created exceeds 128(Which is the total number of subtended ports supported), the config returns an error. Also, the latest modules created should be deleted first, so, for instance if module 1, 2 and 3 created, the deletion has to be in the sequence 3, 2 and 1, the exception to this restriction is when all the modules created are being deleted with a single call to the 'bcm_modid_config_delete_all' API.

Parameters
[in,out]unitUnit number.
[in,out]modidModule information
Return values

Create mapping from local gport id to system gport id.

Creates mapping from local gport id to system gport id. The following flags are available:

Stack Flags
bcmStkModuleMHPriorityUse Assign the priority and color to the MH header and utilize the same for queuing purposes. This will be used for HG Proxy and two-pass loopback applications. For the HG Proxy, the color/priority can be assigned at the Ingress device and consumed at the Egress device for queuing purposes, in this case this control needs to be set at both the devices. For a two-pass loopback solution, the priority/color can be assigned in pass one and used in pass two, in such a case this control will only be set on the local device. This control is only applicable on these XGS devices: bcm5634x, bcm5663x, bcm5644x, bcm5645x, bcm5684x, bcm5664x, bcm5685x, bcm5686x and bcm5696x.

BCM_STK_GPORT_MAP_REPLACE Replace existing mapping.

Set, get, or delete Fabric unicast table entries.

These APIs provide control module steering in StrataXGS fabric chips. Each ingress port can be independently programmed to steer packets with differing destination module identifiers to a set of egress ports. Normally the destination port bitmap has only a single destination port set. The CPU port on a fabric device can also be used as a destination. This is the way that a module identifier can be programmed into a board that contains only fabric devices.
