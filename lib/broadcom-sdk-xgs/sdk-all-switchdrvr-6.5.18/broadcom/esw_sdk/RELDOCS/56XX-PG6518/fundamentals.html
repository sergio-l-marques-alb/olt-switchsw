This section describes fundamental concepts needed for understanding the Broadcom BCM API calls. It describes the basic types and function calls for manipulating those types.

The BCM layer explicitly addresses specific devices in a system. This is indicated by the fact that nearly all BCM API calls have unit as the first parameter. This is a small integer indicating the device on which the function should be applied.

When supported, the Dispatch Layer (see bcm_attach) allows the BCM driver to address not only those devices to which the CPU has a direct connection (for example, via a PCI bus), but also those connected to remote CPUs as well. This allows the BCM layer to manage an entire system.

After devices, the most important object is the port. This is illustrated by the fact that the BCM Port API is the largest set of functions by far. The BCM API uses small integers to refer to ports. These are generally the physical port numbers on the device. Thus, it is necessary to know the particulars of the device being updated to address the ports on that device.

All BCM identifiers begin with the prefix bcm_.

Other important notions are typical of Ethernet switches. Here is a list of the typical flow.

  • Packets are received on ports (ingress).
  • Packets pass through ingress logic which analyzes the packet.
  • Packets are conveyed through the switch depending on available resources.
  • Packets are passed to the egress of a different port (egress).

A packet may be dropped if resources are insufficient, or if the packet meets certain conditions which depend on the configuration of the ingress and egress ports. For example, if the packet arrives with a VLAN tag and the ingress port does not belong to that VLAN, the packet may be discarded if "ingress filtering" is enabled on the port.

The packet may be updated in certain cases, especially at egress. For example, if Layer 3 support is enabled and the packet is an IP packet, then the Layer 3 header of the packet may be updated changing the destination Layer 2 address and decrementing the TTL field in the Layer 3 header.

The following is a list of fundamental aspects of switching which are all managed by the BCM API.

  • Layer 2. Switch devices in the system generally inspect the Layer 2 header of a packet, possibly learn the source address and switch the packet based on the destination address. Managing the Layer 2 forwarding data base is an important part of the software.
  • Layer 3. Some devices support examining deeper into the packet to determine the Layer 3 characteristics of the packet, updating the packet and forwarding the packet based on this information. Typically, software is responsible for the learning of Layer 3 routes, but there-after, hardware may be able to forward these packets without software intervention.
  • VLAN. A virtual network to which ports may belong. Ports may maintain state which is VLAN specific. A port may belong to multiple VLANs.
  • STG. Spanning tree groups are groups of VLANs. A port has a forwarding state for each spanning tree group and this is the state used for each VLAN in that group.
  • Mirror. The ability to copy packets at one ingress or egress to another physical port.
  • Trunk. A group of ports treated as a logical link, as per IEEE standard 802.3ad.
  • Multicast Group. A collection of ports which receive packets destined for a particular multicast address.
  • Class of Service. The classification of packets into different categories which are used to determine which hardware resources are used to convey the packets. The particular class chosen will depend on a port's ingress settings. It is usually related to the IEEE 802.1P priority field that appears in the packet, if present.
  • Filtering and Metering. The ability to parse the information in packet and to take actions, including accounting, based on that information.
  • Crossing between Data and Control Planes. In the BCM model, the CPU lies in the Control Plane and does little processing of Data Plane packets. However, it can receive packets from and inject packets into the Data Plane.
  • Module ID. Each switch device is assigned a unique Module ID. These values are used by switch fabrics to direct packets with little ingress processing to the correct device. So this is often called a Fabric ID.
  • Stacking. Creating large systems using interconnections between sub-systems. The result should appear with a single point of management and should convey packets as a single switch or router.

Policy on API Defaults

Broadcom does not guarantee default values set within the SDK and reserves the right to make changes to default values in between releases.

If a particular default value is required for the application software to work properly, it must be explicitly set by the user application.

Port Bitmaps

Many BCM function calls use a port bitmap to describe a collection of ports for a particular device. This is the type bcm_pbmp_t.

Some devices maintain more than 32 ports, so the exact definition of a port bitmap depends on the devices supported. Consequently, the application should only use the macros described below to manipulate bitmaps. Using standard C operators such as "|" and "&" may will result in compile errors or in unpredictable behavior. This includes using comparison operators such as "==" and "!=".

As with any macro definition, care must be taken to avoid side affects. Only simple C variables should be passed as parameters to ensure proper results and future compatibility.

BCM_PBMP_CLEAR

BCM_PBMP_ASSIGN

BCM_PBMP_NEGATE

Port bitmap assignment operators.

BCM_PBMP_CLEAR initializes a port bitmap to the empty set. ASSIGN and NEGATE macros either copy the src_pbmp to the dst_pbmp or copy the bitwise negation of src_pbm to dst_pbmp, respectively. It is legal to use BCM_PBMP_NEGATE(pbmp, pbmp) to operate on the source.

Synopsis

#include <bcm/types.h>
#define BCM_PBMP_CLEAR(dst_pbmp)
#define BCM_PBMP_ASSIGN(dst_pbmp, src_pbmp)
#define BCM_PBMP_NEGATE(dst_pbmp, src_pbmp)
Parameters
dst_pbmpPort bitmap containing result of operation
src_pbmpPort bitmap containing source of operation

BCM_PBMP_AND

BCM_PBMP_OR

BCM_PBMP_XOR

Bitwise logical operations on port bitmaps.

Provides bitwise AND, OR, and XOR operations on two-port bit maps. pbmp_lhs is ended with pbmp_rhs, the result is placed in pbmp_lhs.

These functions are implemented as macros, so take care not to use parameters that have side affects.

Synopsis

#include <bcm/types.h>
#define BCM_PBMP_AND(pbmp_lhs, pbmp_rhs)
#define BCM_PBMP_OR(pbmp_lhs, pbmp_rhs)
#define BCM_PBMP_XOR(pbmp_lhs, pbmp_rhs)
Parameters
pbmp_lhsLeft hand side of logical and operation, contains result on completion
pbmp_rhsRight hand side of logical AND operation

BCM_PBMP_PORT_ADD

BCM_PBMP_IS_NULL

BCM_PBMP_NOT_NULL

BCM_PBMP_EQ

Compare port bitmaps.

Check for the existence of any port in the bitmap (BCM_PBMP_NOT_NULL), no ports in the bitmap (BCM_PBMP_IS_NULL), or compare two bitmaps for equality (BCM_PBMP_EQ and BCM_PBMP_NEQ).

pbmp_bRight hand of compare

BCM_PBMP_ITER

BCM_PBMP_COUNT

Iterate over or count ports in a port bitmap.

BCM_PBMP_COUNT simply counts the number of ports in the port bitmap.

BCM_PBMP_ITER is a C Code macro that helps construct a loop to iterate through all ports specified in a port bitmap. It is invoked similar to a for loop as shown below and it accepts either a simple C statement or a C block encompassed by braces.

/*
* Simple iterate example that prints out the port number of all ports in a
* Port Bitmap (bitmap)
*/
void
sample_pbmp_iter(bcm_pbmp_t bitmap)
{
bcm_port_t port;
BCM_PBMP_ITER(bitmap, port) {
printf("bitmap contains port %d", port);
}
}

Synopsis

#include <bcm/types.h>
#define BCM_PBMP_ITER(pbmp, port)
#define BCM_PBMP_COUNT(pbbm, count)
Parameters
pbmpPort bitmap to inspect
portC variable assigned the current port inside body of loop; the parameter port must be of type bcm_port_t
countC variable assign count of number of ports specified in the port bitmap; the parameter count is type int

GPORT Utility Macros

Traditionally, APIs have had multiple parameters to specify destination information (for example, module, port, or trunk). The GPORT (generic port) type was introduced to provide a single 32-bit identifier for destinations (see API_BASIC_TYPES). GPORT can be used to uniquely identify the following destination types:

  • Local port number (a physical port on the local unit)
  • {module, port} pair
  • Trunk identifier
  • Black hole (packets are to be dropped)
  • Local CPU (packets are to be directed to local CPU)
  • L2 MPLS virtual-port (VPLS/VPWS)
  • Subport group
  • Subport virtual-port
  • MAC-in-MAC virtual-port
  • WLAN virtual-port
  • CoSQ Scheduler
  • Tunnel identifier
  • {device ID, port} pair (ports on devices without a module ID)
  • Child port on a parent GPORT
  • Egress only group of FIFOs
  • Egress only child port on a parent GPORT
  • Egress only {module, port} pair
  • System Port ID in SAND Architecture

Referring to DNX only, note that, starting with Jericho, a large number of MACROs has been added a 'core' paerameter to indicate the core to which GPORT relates. The names of the newly introduced MACROs is, in most case, the same as existing MACRO with 'CORE' added before the operation. For example, BCM_COSQ_GPORT_COMPOSITE_SF2_SET and its corresponding BCM_COSQ_GPORT_COMPOSITE_SF2_CORE_SET. When such a pair exists, the former is equivalent to the latter except that the 'core' parameter is considered as 'all cores'.

}

Port Mapping

With the increased complexity and performance of the Broadcom switch devices, there is no longer always a direct mapping between the port numbers used by the software programming interface and the physical port pin-outs on the switch chip. This is particularly a problem when a design replaces legacy switch chip with a latest pin-compatible switch chip. The register abstraction provided by the BCM API combined with pin-compatibility should ideally make the application migration from the old to the new device straight-forward, however, if the API port maps are different between the old and the new device, many application functions will break.

to address such differences in the API port maps, the BCM API supports port remapping within the API itself. As an example it is possible to make the port map of a BCM56620 device look like the port map of a BCM56504 device.

The port mapping tables are selected and customized through the configuration manager, and no API functions are provided for changing these tables.

By default, BCM API port mapping is not compiled into the SDK libraries. A detailed description of how to enable and configure BCM API port mapping is outside the scope of this manual. Consult the relevant application notes for further information.

VNTAG/ETAG

VNTAG information is added in the Ethernet packet to process packets in a NIV-Enabled Network, and ETAG information is added in the Ethernet packet to process packets in a PE-Enabled Network.

VNTAG_ETAG_TYPES shows the types that are designated for working with VNTAG/ETAG.

VNTAG/ETAG types
Type Description
bcm_vntag_t

A structure that includes

  • direction -Direction (0-IV to VIS, 1-VIS to IV)
  • pointer -Pointer (0-Unicast, 1-Multicast)
  • dst_vif -Destination Virtual Interface
  • loop -Looped (0-No loop, 1-Loop)
  • src_vif -Source Virtual Interface
bcm_vntag_action_t

An enum that includes

  • bcmVnTagActionNone -No action
  • bcmVnTagActionChange -Replace VnTag if already present or add VnTag if not already present
  • bcmVnTagActionDelete -Delete VnTag
  • bcmVnTagActionCount -Always Last, not a usable value

  • pcp -Priority code point
  • de -Discard eligibility
  • src _vid-Source extended port vlan id
  • vid -Extended port vlan id
