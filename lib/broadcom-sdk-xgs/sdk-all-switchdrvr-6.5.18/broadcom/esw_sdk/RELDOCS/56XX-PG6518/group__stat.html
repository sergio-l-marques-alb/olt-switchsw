Creates a {counting_source, counting_mask} to lif_stack_level mapping. For Issuing a counter command with the the LIF in lif_stack_id_to_count in the packet LIF stack.

An Out-LIF/In-LIF value is one of the parameters used for a decision if a counting action is valid. There can be two valid ranges of Out-LIF/In-LIF: range 0 and range 1. For ingress PP (IRPP) there are 2 counters, counter pointers are calculated as follows:

  • At this point there are up to four indexes into the LIF table: the first is the outer-most LIF; the remaining ones have a 2-bit index/valid attribute.
  • The LIFs are ordered according to their index (we assume that indexes are not identical)
  • Each LIF is compared to two ranges (Cfg-Counter-In-LIF-Range-0|1). The comparison result is: 2'b00 ? LIF is invalid 2'b01 ? LIF is not in any range 2'b10 ? LIF is inside range '0' 2'b11 ? LIF is inside range '1'
  • The value {Outer-most-LIF-Range, LIF-0-Range, LIF-1-Range, LIF-2-Range} is mapped to Counter-0|1-Source which can have the following values: 2'b00 ? Outer-most-LIF 2'b01 ? LIF-0 2'b10 ? LIF-1 2'b11 ? LIF-2
  • Counter-0|1-Pointer is determined according to Counter-0|1-Source (padded with 5 zeros}; Counter-0|1-Update is set if the LIF indicated by Counter-0|1-Source is valid. Egress EPNI, At this point the last three Out-LIFs in the encapsulation stack are available; not all are valid. Each LIF is compared to two ranges (Cfg-Counter-Out-LIF-Range-0|1). The comparison result is: 2'b00 ? LIF is invalid 2'b01 ? LIF is not in any range 2'b10 ? LIF is inside range '0' 2'b11 ? LIF is inside range '1'
  • The value {Counter-0-Mode equal LIF (1), Counter-1-Mode equal LIF (1), LIF-0-Range (2), LIF-1-Range (2), LIF-2-Range (2)} is mapped to Counter-0|1-Source which can have the following values: 2'b00 ? Invalid (i.e. NULL) 2'b01 ? LIF-0 2'b10 ? LIF-1 2'b11 ? LIF-2
  • If Counter-0|1-Mode is LIF
  • Counter-0|1-Pointer[15:0] is calculated according to Counter-0|1-Source
  • Counter-0|1-Valid is set if Counter-0|1-Source is not Invalid
  • If Counter-0|1-Valid then
  • {Counter-1-Pointer[20:16], Counter-0-Pointer[20:16]} = {Egress-MC(1), System-MC(1), Out-PP-Port(8)}
  • Counter-0|1-Pointer += Cfg-Counter-0|1-Base[Counter-0|1-Mode](21) + Out-PP-Port.Counter-0|1-Pointer-Base(21)

Configure a LIF-counting-profile - define it's range - and what counter-command collects it.

An Out-LIF/In-LIF value is one of the parameters used for a decision if a counting action is valid. There can be two valid ranges of Out-LIF/In-LIF: range 0 and range 1. For ingress PP (IRPP) there are 2 counters, counter pointers are calculated as follows:

  • At this point there are up to four indexes into the LIF table: the first is the outer-most LIF; the remaining ones have a 2-bit index/valid attribute.
  • The LIFs are ordered according to their index (we assume that indexes are not identical)
  • Each LIF is compared to two ranges (Cfg-Counter-In-LIF-Range-0|1). The comparison result is: 2'b00 ? LIF is invalid 2'b01 ? LIF is not in any range 2'b10 ? LIF is inside range '0' 2'b11 ? LIF is inside range '1'
  • The value {Outer-most-LIF-Range, LIF-0-Range, LIF-1-Range, LIF-2-Range} is mapped to Counter-0|1-Source which can have the following values: 2'b00 ? Outer-most-LIF 2'b01 ? LIF-0 2'b10 ? LIF-1 2'b11 ? LIF-2
  • Counter-0|1-Pointer is determined according to Counter-0|1-Source (padded with 5 zeros}; Counter-0|1-Update is set if the LIF indicated by Counter-0|1-Source is valid. Egress EPNI, At this point the last three Out-LIFs in the encapsulation stack are available; not all are valid. Each LIF is compared to two ranges (Cfg-Counter-Out-LIF-Range-0|1). The comparison result is: 2'b00 ? LIF is invalid 2'b01 ? LIF is not in any range 2'b10 ? LIF is inside range '0' 2'b11 ? LIF is inside range '1'
  • The value {Counter-0-Mode equal LIF (1), Counter-1-Mode equal LIF (1), LIF-0-Range (2), LIF-1-Range (2), LIF-2-Range (2)} is mapped to Counter-0|1-Source which can have the following values: 2'b00 ? Invalid (i.e. NULL) 2'b01 ? LIF-0 2'b10 ? LIF-1 2'b11 ? LIF-2
  • If Counter-0|1-Mode is LIF
  • Counter-0|1-Pointer[15:0] is calculated according to Counter-0|1-Source
  • Counter-0|1-Valid is set if Counter-0|1-Source is not Invalid
  • If Counter-0|1-Valid then
  • {Counter-1-Pointer[20:16], Counter-0-Pointer[20:16]} = {Egress-MC(1), System-MC(1), Out-PP-Port(8)}
  • Counter-0|1-Pointer += Cfg-Counter-0|1-Base[Counter-0|1-Mode](21) + Out-PP-Port.Counter-0|1-Pointer-Base(21) In Jericho there is a local-to-global-LIF-mapping. Users do not control the local-LIF-allocation. To configure the LIF counting functionality we need to control from where a local-LIF is allocated. The configuration of local-LIF-allocation is decided using this API:
  • A profile is mapped to the IRPP/ETPPXrange1/0.
  • The two ranges must be mutauly exclusive.
  • Lif-counting-profile configuration can be call only prior to any LIF allocation in the system, else we returns an error.
  • First 16 local-LIFs are reseved, cannot be counted. Configuring the counter processor:
  • The EPNI and IHP have two counter commands each,
  • configure the counter processor as such that counter-command 0 counts LIFs in range 0, and counter-command 1 count LIFs in range 1.
  • If we have two LIFs in range x on a packet (which should not happen) we have a new API indicating witch counter to take.
  • Setting the source base_offset will and engine write-enable value ? is defined by the API call. Today the range-size/offset is equal to the profile-size/offset. In the future the range will increase in new allocations.
  • Both cores are configured symmetrically. Values for the pointer-range given in API bcm_stat_counter_config_set() handled acorgdingaly.
