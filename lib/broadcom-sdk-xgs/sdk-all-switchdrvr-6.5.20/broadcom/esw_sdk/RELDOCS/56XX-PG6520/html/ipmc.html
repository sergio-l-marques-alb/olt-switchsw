IP Multicast

Internet multicast applications such as multimedia conferencing, corporate communications, distance learning, distribution of software, stock quotes and news are becoming increasingly popular.

IP Multicast (IPMC) delivers IP traffic from one source to multiple receivers without adding any additional burden on the source or the receivers while using the least possible network bandwidth. The source sends a single stream of packets, which are replicated in the network by IP Multicast-enabled routers only when needed, thus conserving bandwidth. When the packet reaches the LAN to which the receiver hosts are attached, the switch sends out as many copies of the packet as there are ports to which receiver hosts are connected. The L2 MAC address used is attained through IPMC to 802.3 address mapping.

IP Multicast is based on the concept of a group specified by a Multicast address. An arbitrary group of IP hosts interested in receiving a particular data stream must join the group using IGMP. In other words, a Multicast address specifies an arbitrary group of IP hosts that have joined the group and want to receive traffic sent to this group.

The Internet Assigned Numbers Authority (IANA) controls the assignment of IP multicast addresses. It has assigned the old Class D address space to be used for IP multicast. All IP multicast group addresses fall in this range of 224.0.0.0 to 239.255.255.255, of which 224.0.0.0 through 224.0.0.255 are reserved for network protocols. 224.0.1.0 through 238.255.255.255 are called globally-scoped addresses which can be used to multicast data between organizations and across the Internet. 239.0.0.0 through 239.255.255.255 contains limited/administratively scope addresses, which are constrained to a local group or organization (see RFC 2365).

Broadcom StrataXGS devices provide efficient IP multicast capability. The BCM IPMC APIs allow applications to manage the IPMC features and related tables/registers for all BCM56xxx devices, without much knowledge of the switch chip internals.

The bcm_ipmc_* APIs should not be used if the bcm_multicast_* APIs are employed. The multicast module supersedes the previous IPMC modules.

Data Types Defined for IPMC APIs

  • bcm_ipmc_addr_t contains all relevant information for setting up an IPMC group. It is passed to bcm_ipmc_xxx functions.
  • lookup_class is the class identifier. This is a class ID number supplied by the application to group multiple IPMC entries for classification purposes.
  • ing_intf represents L3 ingress interface associated with this entry. Starting from BCM56860 device family ing_intf should be used to represent L3 ingress interface as oppose to vid used on previous devices. vid or ing_intf is processed based on BCM_IPMC_L2 flag. Also refer soc property ipmc_l2_use_vlan_vpn which can be used in combination of vid and ing_intf to utilize IPMC L2 broadcast domain lookup capability of device.

BCM_IPMC_RP_ID_EXPECTED_INTF_RESOLVE

Indicates configuration resolution between rendezvous point ID (l3a_rp) and L3 interface (l3a_expected_intf) is not successful during route get/find/traverse APIs. bcm_l3_route_t has l3a_rp and l3a_expected_intf members set without BCM_IPMC_POST_LOOKUP_RPF_CHECK in l3a_ipmc_flags. BCM_IPMC_DISCARD Indicates to discard packets on entry hit.

Common IPMC APIs

bcm_ipmc_flexctr_object_set(int unit, bcm_ipmc_addr_t *info, uint32_t value); Set flex counter object value for the given IPMC group.
bcm_ipmc_flexctr_object_get(int unit, bcm_ipmc_addr_t *info, uint32_t *value); Get flex counter object value for the given IPMC group.

bcm_ipmc_stat_detach(int unit, bcm_ipmc_addr_t *info); Detach counter entries to the given IPMC group.
bcm_ipmc_stat_id_get(int unit, bcm_ipmc_addr_t *info, bcm_ipmc_stat_t stat, uint32 *stat_counter_id) Get stat counter id associated with given IPMC group.
