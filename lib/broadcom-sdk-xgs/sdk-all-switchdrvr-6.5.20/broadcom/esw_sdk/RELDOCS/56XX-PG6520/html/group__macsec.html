Get current event notification status for the specified event.

The bcm_macsec_event_enable_get API returns the current enable or disable status for the specified event. If the returned value in the enable parameter is zero, this indicates that the MACSec event notification for the event is disabled. If the returned value in the enable parameter is one, this indicates that the event handling for the specified event is enabled.

Enable or disable event notification for the specified event.

The bcm_macsec_event_enable_set API enable or disables the event notification for the specified event. If the enable parameter is zero, this will disable the MACSec event notification to applications. If the enable parameter to the API is one, this will enable the MACSec event notification to the user applications. bcm_macsec_event_t enum identifies the event type and has the following definition:

typedef enum {
BCM_MACSEC_EVENT_PRE_KEY_EXPIRE = 0, // Soft expiry
BCM_MACSEC_EVENT_KEY_EXPIRED, // Assoc expired
BCM_MACSEC_EVENT_KEY_RENEWED, // Assoc created
BCM_MACSEC_EVENT_HW_FAILURE, // HW failure
BCM_MACSEC_EVENT__COUNT
} bcm_macsec_event_t;

A summary of various events generated by the MACSec software module is defined in XREF_MACSEC_EVENT_v .

BCM_MACSEC_EVENT_KEY_RENEWED Triggered whenever a new secure association is created or updated

Register application event handler function.

The bcm_macsec_event_register registers the user-provided callback function for event handling with the MACSec module. This callback is invoked by the MACSec software module in response to an event being detected. When multiple handlers are registered for event notification, all the handlers are invoked by the MACSec software. Provide the event handler callback function pointer conforming to the following signature:

Create/update a MACSec flow object.

bcm_macsec_flow_create creates a MACSec flow on Broadcom MACSec devices. A flow specifies how to handle packets which have the packet fields matching the ones defined by this flow. Each flow has a corresponding action object (defined by bcm_macsec_flow_action_t structure). Actions to the flow specify the actions/operations/packet modifications that need to be carried on the packets belonging to the flow. In egress directions, packets can come to MACSec core either tagged (having a SECTAG header) or could be untagged (No SECTAG header). For the packets that are coming untagged, the decision as to the packet needs to be forwarded to Controlled port Vs Uncontrolled port Vs No-Port (that is, DROP) is made by the egress flows. If the Egress flow's action is set to BCM_MACSEC_FLOW_ACTION_CONTROLLED_FORWARD, packets are forwarded to egress controlled port. Also if the action is BCM_MACSEC_FLOW_ACTION_CONTROLLED_FORWARD, the flow's action must specify a valid secure channel to which the flow belongs to, because that's where the information regarding whether encryption required or not, PN number, encryption key etc come from. For Egress controlled flows, user can control various fields of the SECTAG header, for example the SECTAG.C, SECTAG.E bits etc. by using flags like BCM_MACSEC_FLOW_ACTION_SECTAG_E_VALUE_0, BCM_MACSEC_FLOW_ACTION_SECTAG_E_VALUE_1 etc, which in turn determine whether the packet needs encryption and/or protection. If none of the SECTAG.C and SECTAG.E action flags are set, API will set the SECTAG.C and SECTAG.E bits according to the properties of the channel. If the flow's action is set to BCM_MACSEC_FLOW_ACTION_UNCONTROLLED_FORWARD, packets are forwarded to Uncontrolled port. If the flow's action is set to BCM_MACSEC_FLOW_ACTION_DROP, packets will be dropped. In the ingress direction, all the packets by default go to Controlled port unless they are directed to Uncontrolled port. The decision to forward packet to Uncontrolled port is made by the ingress flows that have action set to BCM_MACSEC_FLOW_ACTION_UNCONTROLLED_FORWARD. The flow's match fields specify the signature of the packets that matches the flow. These flows typically program the prefilter table in hardware. Besides these kind of flows (Ingress prefilter flows), user can create flows in ingress direction which take effect after packets have been destined for Controlled or Uncontrolled port. For Uncontrolled port, these flow provide extra actions which can be taken once the packets are destined to Uncontrolled port for example to selectively drop, redirect the packets etc. For controlled port, these flows are matched after SECY processing is done that is, can match on packet fields which were otherwise encrypted coming in from line. Also these ingress controlled port flows can take actions on various SECY error conditions like secure channel not found etc. For these flows, the user must specify BCM_MACSEC_FLOW_MATCH_CONTROLLED and/or BCM_MACSEC_FLOW_MATCH_UNCONTROLLED in bcm_macsec_flow_match_t structure's flag member. If none of these flags in set for these flow, default is BCM_MACSEC_FLOW_MATCH_CONTROLLED. The user must provide a valid bcm_macsec_flow_match_t structure which specifies the match criteria for the flow, and also a valid bcm_macsec_flow_action_t structure which specifies the corresponding actions to be taken on the match. The API will return flow identifier in flowId parameter on successful flow creation. User can update the flow using the same API. The flags parameter to the API specify whether to create a new flow or update an existing flow. If the flag BCM_MACSEC_SECURE_REPLACE is set, then this is essentially an update operation. In this case if the flow identified by flowId, does not exists, an error is returned, otherwise the flow's bcm_macsec_flow_match_t and bcm_macsec_flow_action_t are updated. If BCM_MACSEC_SECURE_WITH_ID is specified, this will create a flow with the specified flowId if it already does not exist. If the flag BCM_MACSEC_SECURE_REPLACE is set, then this is essentially an update operation. In this case, if the flow object does not exist, an error is returned, otherwise the flow object and/or its corresponding action object is updated. If BCM_MACSEC_SECURE_WITH_ID is specified, this will create a flow with the specified flowId if it already does not exist. If neither of the flags (BCM_MACSEC_SECURE_REPLACE or BCM_MACSEC_SECURE_WITH_ID) are set, then a new flow is created and the flowId for the newly created flow is returned in the flowId parameter. bcm_macsec_flow_match_t structure specifies the fields of the packet for which the lookup is done. Create a MACSec Secure association for the specified secure channel.

bcm_macsec_secure_assoc_create creates a secure association for the specified secure channel. Each secure association object created is bound to a secure channel. The flags parameter controls the operation of the API, if the BCM_MACSEC_SECURE_WITH_ID flag is set in the flags parameter, a secure association with the user-specified assocId is created if it already does not exist. If secure association already exists, an error BCM_E_EXISTS is returned. If the flag parameter contains BCM_MACSEC_SECURE_REPLACE, it updates secure association that already exists. The assocId parameter specifies the ID for the secure association in this case. If the association does not exist, an error BCM_E_NOT_FOUND is returned. If the flag parameter is None (that is, None of the BCM_MACSEC_SECURE_WITH_ID or BCM_E_REPLACE is set), then a new secure association object is created and a new assocId is assigned. The assocId in this case returned back to the caller in assocId. bcm_macsec_secure_assoc_t structure provides the attributes for the secure association.

Create a MACSec secure channel.

bcm_macsec_secure_chan_create creates a MACSec secure channel on Broadcom MACSec devices. The secure channel created is simplex channel, that is, either its associated with the ingress Direction or the egress direction. The direction of the channel is controlled by BCM_MACSEC_SECURE_CHAN_INGRESS flag (Default is assumed to be egress direction). All the secure channels explicitly require that the secure channel identifier (SCI) be specified in the bcm_macsec_secure_chan_t structure. The flags parameter to the bcm_macsec_secure_chan_create controls the behavior between channel creation versus channel update. If the flag is BCM_MACSEC_SECURE_REPLACE, the specified channel is updated, otherwise if it is not set a new channel is created. A valid SCI must be specified in the bcm_macsec_secure_chan_t structure. The flags parameter in the bcm_macsec_secure_chan_t controls various attributes for the channel as described in XREF_MACSEC_SECURE_CHAN_v . If BCM_MACSEC_SECURE_WITH_ID is specified in the flags parameter, it will create a secure channel with the specified chanId if already does not exist. If none of the flag (BCM_MACSEC_SECURE_REPLACE or BCM_MACSEC_SECURE_WITH_ID) is set, then a new channel is created and the channel-ID for the channel is returned in the chanId parameter. bcm_macsec_secure_chan_t structure specifies the attributes of the secure channel.

The user must always specify a valid SCI in the bcm_macsec_secure_chan_t structure. The flags parameter in the bcm_macsec_secure_chan_t controls various attributes for the channel. The summary of the various flags that can be set for the flag is defined in XREF_MACSEC_SECURE_CHAN_v .
