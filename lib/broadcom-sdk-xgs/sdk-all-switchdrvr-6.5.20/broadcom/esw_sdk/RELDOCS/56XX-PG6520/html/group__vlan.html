void bcm_vlan_action_set_t_init ( bcm_vlan_action_set_t *  action)

Initialize a VLAN tag action set structure.

Initializes a VLAN tag action set structure to default values. This function should be used to initialize any VLAN tag action set structure prior to filling it out and passing it to an API function. This ensures that subsequent API releases may add new structure members to the bcm_vlan_action_set_t structure, and bcm_vlan_action_set_t_init will initialize the new members to correct default values. StrataXGS?III devices that support the VLAN Action APIs can specify actions for a packet's inner and outer VLAN tag (add, delete, replace, copy, or do nothing). Depending on the number of VLAN tags in a packet along with the TPID value(s), a packet can be described as double-tagged, outer-tagged, inner-tagged, or untagged. In addition, VLAN tags with VLAN ID equal to 0 are referred to as 'priority-tagged'. Separate VLAN actions can be specified each of these types. int bcm_vlan_gport_add ( int  unit,
bcm_vlan_t  vlan,
bcm_gport_t  port,
int  flags 
)

Add a virtual or physical port to the specified VLAN.

Adds the given port to the VLAN. For BCM56524 and BCM56634, the port can be either a WLAN virtual port or a regular physical port. For BCM56840, the port can be a layer 2 logical port or a regular physical port. WLAN virtual ports or layer 2 logical ports that are members of a VLAN will receive broadcast, multicast and unknown unicast packets that are flooded to the VLAN. For applications that require different recipients for broadcast, unknown multicast and unknown unicast, the different multicast groups can be set using the bcm_vlan_control_vlan_set API. Virtual Port(VP) can be added to the vlan through this API to obtain the vlan membership. Due to the large amount of virtual port number, unlike physical port, switch hardware can not uses a port bitmap in the vlan table entry to represent the vlan membership. Instead the hardware uses two different methods: VP group and hash table to provide VP vlan membership. The hash table method is only available on bcm56850 and later switch devices. The VP group referring as indirect vlan membership method is to associate a group of virtual port with a VP group and then the VP group establishes the membership with the vlan. The hardware uses the VLAN_MEMBERSHIP_PROFILE field in the SOURCE_VP table to associate a virtual port with the VP group, and use the VP_GROUP_BITMAP field in the vlan table to establish the vlan membership with the VP group. This method works well when a group of VPs belong to the same number of vlans such as in the following case:

VP group1 with members VP1, VP3, VP5 has the same membership with vlan 1 to vlan 1000
VP group2 with members VP2,VP4, VP6 has the same membership with vlan2 to vlan 1000

SDK software provides two different mechanisms to utilize this method. The first and default one refers to as SDK managed VP group. In this mechanism, the VP group assignment and vlan association is completely hidden from user and is automatically done by SDK software when user calls bcm_vlan_gport_add() API without BCM_VLAN_GPORT_ADD_VP_VLAN_MEM BERSHIP flag. Note that for this usage model, user must first call bcm_port_vlan_member_set() API with BCM_PORT_VLAN_MEMBER_INGRESS or BCM_PORT_VLAN_MEMBER_EGRESS flag set prior to calling bcm_vlan_gport_add() API. It offers convenience and transparency to the user. However it may not efficiently utilize the limited number of VP groups when user calls bcm_vlan_gport_add() API in certain orders. In the worst case, user may see the VP group resource gets exhausted very quickly. To combat this problem, software offers an alternative referring as unmanaged VP group mechanism. In this mechanism, user takes control of VP group assignment by calling bcm_port_control_set() with bcmPortControlVlanVpGroupEgress/Ingress types, and vlan association by calling bcm_vlan_gport_add() API with VP_GROUP type gport. User should be aware these two mechanisms are mutually exclusive. In order to use the un-managed mechanism, user needs to do one of the above two callings first once to allow it disable the managed mechanism before it starts. The hash table method referring as direct vlan membership method uses a hash table. The key of the each hash table entry is the virtual port number and a vlan number. Thus it directly establishes VP vlan membership with the vlan. In addition to vlan membership , this method also provides Spanning Tree state control on per VP base. Otherwise it was controlled by the physical port where the virtual port resides. User can call bcm_vlan_gport_add() with flag BCM_VLAN_GPORT_ADD_VP_VLAN_MEMBERSHIP to utilize this method. The indirect vlan membership method and direct vlan membership method can be used at the same time. To maximize the resource, user should carefully choose the suitable method to program the virtual port vlan membership. DO_NOT_ADD flags can be used to update multicast group membership. Calling bcm_vlan_gport_add() with DO_NOT_ADD falgs, port is removed from multicast group, otherwise it is added to multicast group. Note: Flag BCM_VLAN_GPORT_ADD_INGRESS_ONLY and BCM_VLAN_GPORT_ADD_EGRESS_ONLY have different meaning when used in VP group and hash table methods. For VP group, when BCM_VLAN_GPORT_ADD_INGRESS_ONLY is set, SDK will add the VP group to ingress VLAN's vp_group_bitmap and when BCM_VLAN_GPORT_ADD_EGRESS_ONLY is set, SDK will add the VP group to the egress VLAN's vp_group_bitmap. For hash table, when BCM_VLAN_GPORT_ADD_INGRESS_ONLY is not set, SDK will add the VP to the egress VP-VLAN membership table and when BCM_VLAN_GPORT_ADD_EGRESS_ONLY is not set, SDK will add the VP to the ingress VP-VLAN membership table. Note: Flag BCM_VLAN_GPORT_ADD_INGRESS_ONLY and BCM_VLAN_GPORT_ADD_EGRESS_ONLY have different meaning when used in VP group. These two flags are only available under unmanaged VP group management mechanism.

Add or remove a protocol-based VLAN based on frame type and ether type. The ether type is a 16-bit value that matches the EtherType field in the received packet.
