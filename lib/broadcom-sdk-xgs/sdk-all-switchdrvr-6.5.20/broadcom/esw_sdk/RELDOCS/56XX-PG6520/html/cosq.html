Class of Service Queue Configuration

Class of Service (CoS) Queue Management functions provides for translation of 802.1p priorities into internal class of service queues. It also permits configuration of the scheduling policies used by these queues

The devices supported by the API have different maximum queues, ranging from 1 to 2, 4 or 8 CoS queues. Mapping from 802.1p priorities to these queues is arbitrary. A mapping is an array of bcm_cos_queue_t values indexed by priority.

The order of service of packets in different CoS queues is controlled by the scheduling mode and associated parameters. Weights and delays may be required in some modes. See bcm_cosq_sched_set for more information. Not all devices support all scheduling modes. BCM_COSQ_COMBO describes how to configure the most common modes and combinations.

Scheduling Combinations
Combination Configuration
Strict Priority Call bcm_cosq_port_sched_set with mode argument of BCM_COSQ_STRICT.
Round-Robin Call bcm_cosq_port_sched_set with mode argument of BCM_COSQ_ROUND_ROBIN.
Weighted Round-Robin Call bcm_cosq_port_sched_set with mode argument of BCM_COSQ_WEIGHTED_ROUND_ROBIN. Supply non-zero weights in the weights array argument.
Weighted Fair Queuing Call bcm_cosq_port_sched_set with mode argument of BCM_COSQ_WEIGHTED_FAIR_QUEUING. Supply non-zero weights in the weights array argument. Also, call bcm_cosq_port_bandwidth_set to specify bandwidth parameters for the WFQ.
Strict Priority and Round-Robin Call bcm_cosq_port_sched_set with mode argument of BCM_COSQ_WEIGHTED_ROUND_ROBIN. Supply weights in the weights array argument, with BCM_COSQ_WEIGHT_STRICT for strict priority queues and BCM_COSQ_WEIGHT_MIN for round robin queues.
Strict Priority and Weighted Round Robin Call bcm_cosq_port_sched_set with mode argument of BCM_COSQ_WEIGHTED_ROUND_ROBIN. Supply weights in the weights array argument, with BCM_COSQ_WEIGHT_STRICT for strict priority queues and non-zero values for weighted round robin queues. "port sched get" function may report STRICT for all queues even though they are mixed STRICT and WRR.
Strict Priority and Weighted Fair Queueing Call bcm_cosq_port_sched_set with mode argument of BCM_COSQ_WEIGHTED_FAIR_QUEUING. Supply weights in the weights array argument, with BCM_COSQ_WEIGHT_STRICT for strict priority queues and non-zero values for weighted fair queuing queues. Also, call bcm_cosq_port_bandwidth_set to specify bandwidth parameters for the WFQ. "port sched get" function may report STRICT for all queues even though they are mixed STRICT and WFQ.
Deficit Round Robin Call bcm_cosq_port_sched_set with mode argument of BCM_COSQ_DEFICIT_ROUND_ROBIN. Supply weights in the weights array argument, with BCM_COSQ_WEIGHT_STRICT for strict priority queues and non-zero values in number of kilobytes (1KB = 1024 bytes) for deficit round robin queues. The device supports a few discrete value for weights. The specified value is rounded off to the next higher value supported by the device. BCM_COSQ_WEIGHT_DISCRETE_WEIGHT_LEVELX may be supplied for the weight, indicating weight associated with level X of scheduler element having discrete weight attribute. see BCM_COSQ_Weight_table . "port sched get" function may report STRICT for all queues even though they are mixed STRICT and WDRR.
Note: Queues with 0 weight are always in STRICT mode.

Some devices support Virtual Output Queuing (VoQ) where data is queued at the ingress of the fabric, but queues are organized by output port and class of service (CoS). Certain ports of some devices support Enhanced Transmission Selection (ETS) hierarchical scheduling which requires to use gport version of these APIs to support the functionality. Some devices also support traffic management at a finer granularity and hierarchical scheduling (H-QoS). The gport version of these APIs supports this functionality. As an example, XCore (tm) fabric devices support this functionality. Not all devices may support all the above functionality. Devices BCM5644X/BCM5645X does not support scheduling mode weighted fair queuing.

Queue Congestion Monitoring Application

Queue Congestion Monitoring Overview

Queue Congestion Monitoring is an embedded application that runs on embedded ARM core 0 of Tomahawk(BCM56960) to detect microbursts on links. Microbursts are momentary oversubscriptions that occur on links and last for a very short period of time and hence require constant monitoring to detect. The QCM App on R5 has fast access to counters and TCM memory with low latency, so it can constantly monitor the queues/ports designated by user at a very high rate and summarize the counter data to the user.

The SDK APIs described below achieve configuring of port(s)/queue(s) to be monitored and shared memory host address to R5 to update summarize data.

Event BST Application

Event BST Overview

The BST feature enable applications to set thresholds on a resource and trigger an interrupt whenever the resource usage exceeded the threshold. However, once an interrupt for a resource is triggered, it remain latched on until the SDK clear it. Very often, if multiple entities for the resource cross the threshold in quick succession, only the first one trigger an interrupt while the remaining are lost. Event based BST (EBST) is designed to avoid losing threshold events due to quick bursts of traffic (micro burst).

When EBST is enabled, each threshold breach in the buffer usage results in a threshold event. A hardware process constantly scans all the resources for such threshold events. When an event is detected, the details of the event - timestamp and buffer usage - gets recorded to a FIFO. The SDK is expected to DMA the data from the FIFO and present it to the user.
